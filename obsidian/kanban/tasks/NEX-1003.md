## 🟢 NEX-1003: Galleries table with Supabase S3 image upload and CRUD

Status: Done
Priority: Medium
Source: `obsidian/kanban/specs/backlog/2026-07-10-080836-db-galleries-upload-supabase-s3-26.md`

### Summary
Add a new `galleries` table (Drizzle schema + migration) and a full CRUD flow on the frontend that uploads images to Supabase Storage (S3-compatible) and stores the resulting URL/metadata on each gallery row.

### Context
No Supabase or S3 integration exists in the codebase yet (`src/libs/Env.ts`, `package.json` checked — no `SUPABASE`/`S3` env vars or SDK). This is greenfield: needs new env vars, a storage client lib, schema, validation, API routes/actions, and UI.

### Scope
- `src/models/Schema.ts`: add `gallerySchema` (id, title, imageUrl, imagePath/key, createdAt, updatedAt at minimum).
- `bun run db:generate` migration for the new table.
- `src/libs/Env.ts`: add Supabase S3 env vars (endpoint, region, bucket, access key, secret key, or Supabase project URL + service key — confirm which auth style during planning) validated via Zod.
- New storage lib (e.g. `src/libs/Storage.ts` or `src/libs/SupabaseStorage.ts`) wrapping upload/delete against Supabase S3-compatible storage.
- `src/validations/` Zod schema for gallery create/update (title, image file constraints).
- CRUD surface: list/create/update/delete galleries, with image upload wired to the storage lib. Route group placement (marketing vs dashboard) to be decided in planning.
- i18n: any user-visible strings via `src/locales/{en,fr}.json`.

### Acceptance Criteria
- `galleries` table exists via Drizzle schema + generated migration, applied locally with `bun run db:migrate`.
- User can create a gallery entry by uploading an image; image lands in Supabase Storage (S3-compatible) and the row stores its URL/key.
- User can view, update (replace image/title), and delete gallery entries from the frontend.
- Deleting a gallery entry removes the corresponding object from Supabase Storage.
- All Supabase/S3 credentials read through `Env.ts`, never `process.env` directly.
- `bun run check:types`, `bun run lint`, `bun run check:i18n` pass.

### Validation
- `bun run check:types`
- `bun run lint`
- `bun run check:i18n`
- `bun run test`
- `bun run db:generate` / `bun run db:migrate` succeed locally

### Implementation Plan
See `docs/superpowers/plans/2026-07-10-nex-1003-galleries-supabase-s3.md` and design spec `docs/superpowers/specs/2026-07-10-nex-1003-galleries-supabase-s3-design.md`.

Resolved decisions:
- Supabase auth: service-role key + `@supabase/supabase-js` client (server-side only, never exposed to client).
- Route placement: `/gallery`, behind Clerk auth (not public marketing). Moved off `/dashboard` mid-review since gallery is unrelated to the dashboard page.
- Image constraints: max 5MB, MIME `image/jpeg | image/png | image/webp`, no server-side resize/compress.

### Review Notes (fullstack-guardian pass)

Full-diff fullstack review run (auth/security, upload/storage, server actions, schema, env). Verdict: NEEDS FIX — 2 Important defects found, both fixed:

1. `deleteGallery` deleted the storage object before the DB row — if `db.delete` threw afterward, the row would survive pointing at a missing object (broken image, orphaned row). Fixed: DB row is now deleted first; storage delete runs after and only logs on failure (`src/app/[locale]/(auth)/gallery/actions.ts`).
2. `createGallery` uploaded to storage before the DB insert with no cleanup on insert failure — a failed insert left an orphaned storage object. Fixed: insert failure now triggers a compensating `deleteGalleryImage` call before returning the error state.

A third finding (locale-extraction regex in `src/proxy.ts` not matching bare `/gallery`) was investigated and confirmed to be a non-issue: `localePrefix: 'as-needed'` means the default locale (`en`) is never prefixed, so bare `/gallery` is the correct default-locale path and the unprefixed `/sign-in` redirect is correct — identical, intentional behavior to existing `/dashboard`/`/settings` routes. No change made.

Tests updated to match corrected ordering: `deleteGallery` tests now assert DB-row-first deletion and that storage failures don't block success; added assertion that `createGallery` calls `deleteGalleryImage` cleanup when the DB insert throws.

Re-validated: `check:types` clean, `bun run test` 40/40 pass, `check:i18n` clean. Ready for Done.
